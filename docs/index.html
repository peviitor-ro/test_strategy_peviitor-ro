1. Scope

Platforma peviitor.ro este un motor de căutare open-source a locurilor de muncă din România, construit exclusiv cu tehnologii open-source și gratuite. Scopul acestui document este de a stabili o abordare unificată de testare care să acopere toate componentele majore ale platformei.

⚠️ Always keep in mind: Misiunea principală a platformei este de a oferi utilizatorilor o bază de date agregată, cuprinzătoare și actualizată a tuturor locurilor de muncă din România. Fiecare test, decizie și validare trebuie să servească acest scop.

In Scope

Apache SOLR

Search engine cu două core-uri: job și company. Schemă conform modelelor de date, Basic Authentication.

Indexare

PHP API (api.peviitor.ro)

API BFF cu versiunile v0 (dev/test) și v1 (producție). Simplu, rapid, testabil, deployabil.

API

UI (search-engine)

Interfață React cu Redux, Tailwind CSS, găzduită pe GitHub Pages. Graceful degradation când API-ul e jos.

Frontend

Search Functionality

Filtre: locație, companie, tag-uri, work mode. Rezultate relevante, interfață ușor de folosit.

Căutare

Out of Scope

  • Individual scraper testing (gestionat separat de echipele de scraping)
  • Validator (admin.peviitor.ro) — document separat
  • Mobile application (androidAPP) — document separat
  • orase.peviitor.ro — document separat
  • firme.zira.ro și admin.zira.ro — documente separate
  • splash.peviitor.ro — pagină statică de fallback pe Google Sites, în afara ariei de testare a platformei principale

5. Test Levels

Unit Testing

PHP (API/Core) + JavaScript (React). PHPUnit, Jest, Vitest. La fiecare PR/commit.

Integration Testing

API ↔ SOLR, Frontend ↔ API, Scrapers → Core. Săptămânal, înainte de release.

System Testing (E2E)

Full search flow, admin validation flow. Cross-browser, mobile. Per release.

Regression Testing

Critical path: search, filter, pagination, job detail, company listing. Per release (~weekly).

Acceptance Testing

Validare acceptance criteria, bug reports community, sign-off QA lead. Per release.

7. Test Data

Test data se împarte în 4 categorii:

✅ Valid Data

Job titles existente, city input ("București", "Cluj"), company names, work modes, combined filters, direct search.

❌ Invalid Data

Company < 3 chars, non-existent values, special characters (XSS, SQL injection), invalid city input.

⚡ Edge Cases

Very long input (>500 chars), empty search, case sensitivity, rapid typing, overlapping input, pagination edges.

🔍 No-Result Scenarios

Valid unmatched, over-filtered, expired jobs, invalid URL job.

8. Entry & Exit Criteria

Entry Criteria

  • Code merged to target branch
  • Build passes on CI/CD (GitHub Actions)
  • Frontend builds without errors
  • API returns healthy status
  • SOLR cores accessible with test data
  • Test environment deployed
  • Test cases/checklists updated
  • Known critical bugs fixed or documented

Exit Criteria

  • All S1/S2/S3 bugs fixed and verified
  • Regression test suite: 100% pass rate
  • New features tested and accepted by QA
  • Accessibility: no critical WCAG violations
  • Performance: P95 < 2s search
  • Test summary report reviewed by QA lead
  • Production deploy sign-off from QA lead + project lead
